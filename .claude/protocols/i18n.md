# i18n Protocol

**Applies to**: All products with multi-language or RTL requirements.
**Mandatory for**: connectin (Arabic-first), muaththir (Arabic primary), linkedin-agent (Arabic + English output), any new product targeting Arabic-speaking users.

---

## Core Rules

1. **No hardcoded strings in components.** All user-visible text goes through the i18n system (`next-intl`, `react-i18next`, or equivalent).
2. **Locale files are the source of truth.** English (`en.json`) is always the base locale. Arabic (`ar.json`) is derived from it.
3. **RTL is layout, not a style hack.** Use `dir="rtl"` at the HTML element level, Tailwind RTL utilities (`rtl:ml-4`, `rtl:text-right`), and logical CSS properties (`margin-inline-start` vs `margin-left`). Never use `text-align: right` as an RTL workaround.
4. **Test in both directions.** Every UI component must be visually verified in both LTR and RTL.
5. **Fonts must support Arabic.** Use `Tajawal`, `IBM Plex Arabic`, or `Noto Sans Arabic` for Arabic text. Ensure `font-family` fallback chains are correct.

## Locale JSON Format

```json
{
  "auth": {
    "login": {
      "title": "Welcome back",
      "emailLabel": "Email address",
      "passwordLabel": "Password",
      "submitButton": "Sign in",
      "forgotPassword": "Forgot your password?",
      "errors": {
        "invalidCredentials": "Email or password is incorrect",
        "accountLocked": "Account locked. Try again in {minutes} minutes"
      }
    }
  }
}
```

**Rules:**
- Keys use `camelCase`.
- Nested by feature → component → string.
- Interpolation uses `{variableName}` (compatible with `next-intl` and `react-i18next`).
- Pluralization uses `{count, plural, one {# item} other {# items}}` (ICU format).
- Never embed HTML in translation strings — use rich text APIs instead.

## RTL Implementation

### Tailwind CSS Setup

```js
// tailwind.config.js
module.exports = {
  // Enable RTL variant
  plugins: [require('tailwindcss-rtl')],
}
```

### HTML Dir Attribute

```tsx
// app/layout.tsx (Next.js)
export default function RootLayout({ children, params: { locale } }) {
  return (
    <html lang={locale} dir={locale === 'ar' ? 'rtl' : 'ltr'}>
      <body>{children}</body>
    </html>
  );
}
```

### Logical CSS Properties (prefer these over directional)

| Avoid | Use instead |
|-------|------------|
| `margin-left` | `margin-inline-start` |
| `margin-right` | `margin-inline-end` |
| `padding-left` | `padding-inline-start` |
| `text-align: left` | `text-align: start` |
| `border-left` | `border-inline-start` |
| `left: 0` | `inset-inline-start: 0` |

### Tailwind RTL Utilities

```tsx
// Component example - works in both LTR and RTL
<div className="flex items-center gap-3 rtl:flex-row-reverse">
  <Icon className="shrink-0" />
  <span>{t('label')}</span>
</div>

// Margin
<div className="ms-4">  {/* margin-inline-start: 1rem */}
```

## Arabic Typography

### Font Loading (Next.js)

```tsx
// app/layout.tsx
import { Tajawal } from 'next/font/google';
import { Inter } from 'next/font/google';

const inter = Inter({ subsets: ['latin'] });
const tajawal = Tajawal({
  subsets: ['arabic'],
  weight: ['400', '500', '700'],
  variable: '--font-arabic',
});

// In body className:
// className={`${inter.variable} ${tajawal.variable} font-sans`}
```

### Font Selection in CSS

```css
:root {
  --font-sans: 'Inter', system-ui, sans-serif;
  --font-arabic: 'Tajawal', 'Arial', sans-serif;
}

[dir="rtl"] {
  font-family: var(--font-arabic);
}
[dir="ltr"] {
  font-family: var(--font-sans);
}
```

### Arabic-Specific Considerations

- **Line height**: Arabic script needs more vertical space — use `leading-relaxed` minimum.
- **Letter spacing**: Do NOT use letter-spacing with Arabic. Arabic is a connected script; letter-spacing breaks it.
- **Numbers**: Arabic uses Eastern Arabic numerals (٠١٢٣٤٥٦٧٨٩) in formal contexts. Use `Intl.NumberFormat('ar-SA')` for number formatting.
- **Dates**: Use `Intl.DateTimeFormat('ar-SA', { calendar: 'gregory' })` — avoid Hijri unless explicitly required.

## Locale Detection

```ts
// middleware.ts (Next.js)
import createMiddleware from 'next-intl/middleware';

export default createMiddleware({
  locales: ['en', 'ar'],
  defaultLocale: 'en',        // Override per product: 'ar' for connectin
  localeDetection: true,      // Reads Accept-Language header
});
```

For Arabic-first products (connectin, muaththir):
```ts
defaultLocale: 'ar',
```

---

## Backend Considerations

- Accept-Language header: Fastify should forward or log the locale for analytics.
- Error messages: Return error codes, not English strings. The frontend translates error codes.
- Database: Store content in the user's chosen locale. For user-generated content, store `locale` field alongside content.
- Email: Use locale-specific email templates. Pass `locale` to Email Service.

## Testing i18n

### Unit Tests

```tsx
// Wrap components in i18n provider
import { NextIntlClientProvider } from 'next-intl';
import messages from '@/public/locales/en/common.json';

render(
  <NextIntlClientProvider locale="en" messages={messages}>
    <MyComponent />
  </NextIntlClientProvider>
);
```

### RTL Visual Tests (Playwright)

```ts
// e2e/i18n/rtl.spec.ts
test('login page renders correctly in RTL', async ({ page }) => {
  await page.goto('/ar/login');
  await expect(page.locator('html')).toHaveAttribute('dir', 'rtl');
  // Screenshot comparison
  await expect(page).toHaveScreenshot('login-rtl.png');
});
```

### i18n Check Command

Run `/i18n-check` to validate:
- All locale files have matching keys between `en/` and `ar/`
- No hardcoded strings in component files
- All `dir` attributes correctly set
- Font loading verified

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Using `ml-4` for icon spacing | Use `ms-4` (logical) or `rtl:mr-4` |
| Hardcoding `text-left` | Use `text-start` |
| Embedding HTML in JSON strings | Use rich text component API |
| Applying letter-spacing to Arabic | Remove — breaks connected script |
| Storing translated content in one `content` field | Add `locale` column, or use `contentEn`/`contentAr` |
| Returning English error messages from API | Return error codes; translate on frontend |
